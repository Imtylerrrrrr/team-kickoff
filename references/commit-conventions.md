# 커밋·PR 컨벤션 reference

인터뷰 결과로 규제 수준을 고르고, 사용자가 "왜 이 규칙이야?"라고 물을 때 이 문서로 답한다.

## 규제 스펙트럼 — 어느 수준을 쓸 것인가

| 수준 | 무엇을 규제 | 언제 |
|---|---|---|
| L0 자율 | 없음 (권장 문구만) | 1인 프로젝트 |
| **L1 PR-only (기본값)** | PR 제목만 `타입: 설명` + squash-only. 커밋 자유 | 대회·팀 프로젝트 대부분 |
| L2 풀 Conventional Commits | 모든 커밋 형식 + 훅 | 인터뷰 ④ 자동 릴리즈(semver·체인지로그) 계획이 있을 때만 |

L1이 기본값인 근거:
- squash-only면 main 히스토리 = PR 제목의 나열. 개인 브랜치 커밋은 main에 남지 않는다
- 대회 프로젝트에는 semver도 체인지로그도 없다 — L2의 자동화 이점이 0
- 규제 지점이 하루 커밋 수십 개 → PR 4~5개로 줄어든다
- main 히스토리의 소비자는 리뷰하는 팀원·심사위원·**git log를 읽는 에이전트 자신** — 깨끗한 히스토리는 에이전트 컨텍스트 품질이다

## 상위 프로젝트 실태 (2026-07 조사)

| 프로젝트 | 형식 | 강제 수단 | 시사점 |
|---|---|---|---|
| Linux 커널 | `서브시스템: 요약` + "왜" 본문 | 사람 리뷰 | 50/72 규칙 원조. 형식보다 본문의 설명이 심사 대상 |
| Kubernetes | 타입 접두사 없음. 명령형·50자·마침표 금지 | 리뷰 문화 | 최상위 프로젝트도 타입 형식을 강제하지 않는다 |
| Angular | `type(scope): subject` | commitlint CI | Conventional Commits의 원조 — 목적은 릴리즈 자동화 |

교훈: 형식 규제는 목적(자동화)이 있을 때만 한다. "수준 높은 프로젝트처럼 보여서"는 이유가 아니다.

## 타입 5종 판정 기준

| 타입 | 기준 | 애매할 때 |
|---|---|---|
| feat | 사용자가 체감하는 새 동작 | 체감되는 성능 개선도 feat |
| fix | 잘못된 동작을 바로잡음 | "원래 이래야 했는데"면 fix |
| docs | 문서·주석만 변경 | 코드가 한 줄이라도 섞이면 docs 아님 |
| refactor | 동작 불변, 코드 구조만 변경 | 테스트 추가·정리 포함 |
| chore | 위 넷 다 아님 (설정·의존성·빌드) | 판정에 30초 넘게 걸리면 chore |

11종 풀세트(style·perf·test·build·ci…)를 쓰지 않는 이유: 분류 고민 비용이 분류 가치를 넘는다. 에이전트도 style/refactor/chore 경계에서 흔들리고, 히스토리를 읽는 쪽에 주는 정보 차이는 미미하다.

## PR 제목 작성 규칙

- `타입: 설명` — 콜론 뒤 한 칸, 전체 ~50자
- 명사형/명령형 종결: "로그인 화면 추가" (O) / "로그인 화면을 추가했습니다" (X)
- scope 없음 — `feat(auth):` 금지. 이 팀 규모에서 이득 없이 분류 고민만 추가된다

## 언어 선택 (인터뷰 ③)

판단 기준은 히스토리의 독자다. 국내 대회·전원 한국인 팀이면 한글이 정보 밀도 우위. 국제 공개·글로벌 포트폴리오 목적이면 영어. 에이전트는 어느 쪽이든 잘 쓴다 — 팀이 읽을 언어를 고르면 된다.

## L2 (자동 릴리즈) 선택 시에만

- Conventional Commits v1.0.0 전체 준수 (`BREAKING CHANGE:` 푸터 포함)
- 훅은 husky·commitlint 의존성 대신 무의존 방식: `git config core.hooksPath .githooks` + 셸 스크립트. Node 없는 레포에서도 동작하고, 활성화 명령을 README 3줄 섹션에 명시한다 (조용한 비활성 방지)
- L2에서도 **pre-push main 차단 훅은 넣지 않는다** — 그건 브랜치 보호(또는 소프트 규칙)의 역할이고, 훅 우회 변수를 팀에게 학습시키는 부작용이 강제 이득보다 크다
