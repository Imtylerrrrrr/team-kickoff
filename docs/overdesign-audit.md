# 과설계 감사 기록 (2026-07-03)

> "스킬 자체가 너무 과하게 설계되어 있는지"를 멀티에이전트로 검사한 결과의 보존 기록.
> 실제 수정은 하지 않았고, 나중에 직접 검토하기 위한 아카이브다.
> 후속 판단은 GitHub 이슈로 트래킹한다.

## 방법

- **4개 비평 렌즈**가 각자 "과하다"고 의심되는 지점을 공격적으로 수집 (구조 · 결정 분기 · 투기적 기능 · 개념 무게).
- 각 지적은 **근거 추적 검증자**가 실제 파일 + `docs/red-baseline.md` + `docs/design.md`를 대조해 판정.
- 판정 기준: 복잡성이 **문서화된 baseline 실패 · 리허설 · 실제 GitHub 제약** 중 하나로 추적되면 "근거 있음(방어)", 아무것에도 추적되지 않으면 "과설계 확정".
- 검증자는 Opus 4.8(1M) 20 에이전트. 입증 책임은 비평가에게(추측이라는 증명이 없으면 방어).

## 결과

**16건 의심 → 과설계 확정 0건 / 근거 있어 방어 16건.**

senior 엔지니어가 "오버엔지니어링"이라 부를 부분은 나오지 않았다. 무거워 보이는 장치(3단 폴백·마커·2겹 강제·플레이스홀더)는 전부 실제 GitHub 제약이나 실증된 실패에서 유래했다.

### ⚠️ 이 결과의 해석 주의

검증 기준이 방어 쪽으로 높게 설정돼 있었다(문서 근거가 하나라도 있으면 방어). 이 스킬은 TDD로 실패를 겨냥해 만들었으므로 그 기준이 타당하지만, 그래서 **"근거는 있으나 값어치는 얇은" 부분까지 전부 방어로 넘어갔다.** 아래 경계선 3건은 그 얇은 축에 속하므로 별도로 검토할 값어치가 있다.

## 경계선 3건 (나중에 검토)

### A. L2 경로 (자동 릴리즈) — 가장 많이 공격받음 (2개 렌즈)

- **방어 근거**: baseline 실패 #3이 "릴리즈 계획도 없는데 훅 기본 설치"였고, 인터뷰 ④가 정확히 그 게이트. ④가 없으면 훅을 언제 켤지 판단 불가 → 실패 #4(무조건 최대 세팅) 재발.
- **얇은 지점**: 스킬 자신의 문서가 "대회 팀은 semver·체인지로그 없어 L2 이점 0"이라 명시. 즉 타깃 유저는 ④에 거의 항상 "아니오". **틀린 설계는 아니나 거의 안 쓰이는 분기.**
- **잘라낼 경우 대가**: 릴리즈 하는 정규 팀을 버림 + 실패 #4 재발 위험 + 인터뷰가 4→3문항.
- **잘라낼 경우 이득**: "대회 팀 전용" 정체성에 집중, SKILL.md·commit-conventions·templates 각각 몇 줄 감량.

### B. {{PROTECTION_NOTE}} 3상태 (B·C 동일 출력)

- **방어 근거**: B·C가 같은 문구를 뱉어도 3단 폴백에 1:1. C를 명시 안 하면 에이전트가 "원격 없음"인데 빈 문자열을 뱉는 실수.
- **얇은 지점**: 겉보기 3상태지만 출력 값은 2개. 문서상 "3상태"는 과articulation일 수 있음.
- **잘라낼 경우**: "빈 문자열 / 소프트 문구" 2상태로 단순화 + C 케이스 실수 방지는 산문 한 줄로. 리스크 낮으나 이득도 작음.

### C. en/ + ko/ 전체 이중화

- **방어 근거**: 사용자 명시 결정(영어 주·한글 부, 이중 폴더). 설치는 en XOR ko라 생성된 레포 안에서 규칙이 어긋날 일 없음.
- **얇은 지점**: 규칙 하나 고칠 때마다 두 번 수정하는 유지보수 세금이 실재. 커밋 65a1839·4b508b5가 "en·ko 동기"로 매번 양쪽을 건드림.
- **성격**: 스킬 결함이 아니라 이중 언어 OSS를 위해 감수한 비용. 잘라내려면 단일 소스 + 빌드 타임 번역 생성으로 재구조화해야 하는데, 그건 스킬 범위 밖 인프라.

## 결론

지금 구조를 그대로 둬도 과설계가 아니다. 더 얇게 만들고 싶다면 **후보는 A(L2)** 하나뿐이고, 대가는 위에 적은 그대로다. B는 리스크 낮은 미세 정리, C는 재구조화 비용이 커서 사실상 유지가 합리적.
