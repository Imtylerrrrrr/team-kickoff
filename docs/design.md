# team-kickoff 설계 문서

> 2026-07-03 브레인스토밍 세션에서 확정. 이 문서는 "왜 이렇게 만들었나"의 기록이다.
> 스킬 본문(SKILL.md)과 어긋나면 SKILL.md가 우선 — 단, 어긋남을 발견하면 이 문서를 갱신할 것.

## 정체성

**에이전트 팀을 위한 협업 거버넌스 세팅 스킬.**
대회·프로젝트 시작 시점에 커밋·브랜치·PR 규칙을 인터뷰로 결정하고, 산출물 생성 + GitHub 설정까지 수행한다.

- 대상: 팀원 전원이 코딩 에이전트(Claude Code, Codex, Cursor 등)로 개발하는 팀. 손코딩 팀은 비대상.
- 비범위: 로드맵/구현 계획 생성 (별도 스킬로 분리 — 성격·갱신 주기가 다름), GitLab 등 GitHub 외 호스팅.

## 핵심 설계 결정

### 1. 규칙의 SSOT는 AGENTS.md, 나머지는 한 줄 포인터
- 규칙의 독자 95%가 에이전트다. AGENTS.md는 세션마다 자동 주입되므로 규칙 준수에 사람의 의지가 불필요해진다.
- Codex·Cursor·Copilot 코딩 에이전트는 AGENTS.md 네이티브 직독. Claude Code는 `@AGENTS.md` 임포트 한 줄짜리 CLAUDE.md, Gemini CLI는 한 줄짜리 GEMINI.md 생성.
- **심볼릭 링크 금지**: Windows에서 git 기본 설정(`core.symlinks=false`)이 심링크를 경로 문자열 텍스트 파일로 체크아웃해 깨진다. 반드시 실제 파일로.
- 도구를 인터뷰로 묻지 않고 포인터를 전부 생성한다 (한 줄 파일 비용 ≈ 0, 질문 하나 절약).
- 사람용 가시성 보완: README에 3줄 "협업 규칙" 섹션 (하드 규칙 3개 + AGENTS.md 포인터).

### 2. "PR 규제, 커밋 자율" — squash merge + PR 제목 규칙
- 대회 프로젝트에는 semver·체인지로그 자동화가 없으므로 Conventional Commits의 자동화 이점이 없다. Linux·Kubernetes 등 최상위 프로젝트도 커밋 전체엔 형식을 강제하지 않는다.
- main 히스토리의 소비자는 ① 리뷰하는 팀원 ② 심사위원 ③ 디버깅하는 미래의 팀원 — 그리고 **git log를 컨텍스트로 읽는 에이전트 자신**. 깨끗한 히스토리 = 팀 전원의 에이전트 성능.
- squash-only 머지로 개인 브랜치 커밋은 자유(WIP 허용), PR 제목만 `타입: 설명` 강제. 규제 지점이 하루 커밋 수십 개 → PR 4~5개로 준다.
- 타입은 5종으로 축소: `feat|fix|docs|refactor|chore`. 풀세트 11종은 분류 고민 비용만 늘린다.
- 커밋/PR 언어(한글·영어)는 팀의 선택 — 인터뷰 질문 1개.

### 3. 2겹 강제 구조
- **작성 시점 소프트 강제**: AGENTS.md 규칙 (에이전트가 규칙대로 씀).
- **머지 시점 하드 강제**: GitHub 설정 (squash-only, main 브랜치 보호).
- 어느 팀원의 에이전트가 규칙을 무시해도 main 오염은 GitHub 레벨에서 막힌다. ⑤(도구 파편화)가 불완전 해결이어도 시스템이 안 무너지는 이유.

### 4. 플랜 감지 + 3단 폴백 (검증된 사실)
- **GitHub 무료 플랜 + private 레포에서는 브랜치 보호·rulesets 모두 불가** (공식 문서로 확인, 2026-07). squash-only 머지 설정은 플랜 무관 무료.
- ① public 또는 유료 플랜 → 풀 세팅
- ② private + 무료 → squash-only만 하드 적용, "main 직push 금지"는 AGENTS.md 소프트 규칙으로 강등하고 **이 사실을 사용자에게 명시 보고**
- ③ gh 미설치/미인증/원격 없음 → 사람이 할 일 체크리스트 출력
- **설정 후 `gh api`로 실제 적용을 재확인한 뒤에만 성공 보고** (거짓 성공 보고 금지).

### 5. 멱등성과 기존 레포 안전
- 기존 AGENTS.md/CLAUDE.md는 **절대 덮어쓰지 않는다**. `<!-- team-rules:start/end -->` 마커 섹션으로 최상단 삽입, 재실행 시 마커 사이만 갱신.
- 기존 커밋 히스토리 불가침 — force push 재작성 금지, "지금부터 적용" 원칙.

### 6. 분량 규율
- AGENTS.md 규칙 섹션은 **30줄 상한**. 파묻힘은 위치 문제가 아니라 분량 문제다. 넘으면 설계가 잘못된 것.
- 무거운 것들은 밖에 산다: 검증 체크리스트 → PR 템플릿(GitHub이 자동 삽입), 머지 방식 → 레포 설정, 규칙의 근거 → 스킬 references.

### 7. 인터뷰는 4문항 상한
① 기간·강도 (해커톤 모드 vs 정규 모드) ② 팀 인원 ③ 커밋·PR 언어 ④ 자동 릴리즈 계획 여부.
public/private·플랜·기존 파일은 묻지 않고 감지한다. 해커톤 팀은 세팅에 5분 이상 못 쓴다.

| 페르소나 | 시나리오 | 발견된 실패 | 반영된 수정 |
|---|---|---|---|
| P1 불꽃팀 | 무박 30시간 해커톤 3인, 전원 Claude Code | 새벽에 리뷰어 부재 → 머지 블락 → 규칙 우회 시작 | 해커톤 모드: 승인 0명, 셀프 머지 허용, PR 흐름·제목 규칙은 유지 |
| P2 정공팀 | 3개월 공모전 4인, 도구 혼재, private+무료 | 브랜치 보호 설정 실패 시 에러 사망 또는 거짓 성공 보고 | 플랜 감지 + 3단 폴백 + 적용 재검증 |
| P3 중도합류 | 3주 진행된 레포, AGENTS.md 기존재 | 덮어쓰기로 팀 컨텍스트 파괴 | 마커 섹션 append + 멱등 갱신 |
| P4 제3자 | 스킬을 설치한 외부인 (비한국어, 개인 볼트 없음) | 개인 환경 하드코딩 시 즉사 | 개인(Projects/ 볼트) 연동 완전 제외, 언어는 사용자 따라감 |

## 검증된 외부 사실 (2026-07-03 기준)

- 브랜치 보호/rulesets: public 무료, private은 Pro/Team 이상 ([GitHub Docs](https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-protected-branches/about-protected-branches))
- Claude Code는 AGENTS.md 네이티브 미지원 (요청 이슈 활성, 최신 버전 동작은 상충 정보) → 임포트 포인터 방식은 지원 여부 무관하게 동작
- AGENTS.md는 Linux Foundation 관리 표준, Codex·Cursor·Copilot 코딩 에이전트 등 네이티브 지원
